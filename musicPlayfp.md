# 音频模块复盘：啸叫问题的定位与修复

## 一、问题描述

进入音乐播放状态后：
1. 先录制 2 秒音频 → 可以听到回放（但音量很小）
2. 录音播放完成后 → 产生一个很大的刺耳啸叫，持续不停

---

## 二、问题根因分析

### 2.1 啸叫是怎么产生的？

啸叫的本质是 **声学反馈环路（Acoustic Feedback Loop）**。

我们之前的代码在 `diagRecordThenPlay()` 播放完成后，立即调用了 `startEcho()`。这个函数启动了一个 **实时回环任务（echoTask）**，做的事情是：

```
mic 拾音 → i2s_read() → 数据 → i2s_write() → DAC → PA → 喇叭发声
                                                           │
                    mic 又拾取了喇叭发出的声音 ←─────────────┘
```

这就形成了一个闭合的正反馈环路：

```
┌──→ MIC 拾音 ──→ ES7210 ADC ──→ I2S RX ──→ ESP32 ──→ I2S TX ──→ ES8311 DAC ──→ PA ──→ 喇叭 ──┐
│                                                                                                 │
│                    声波在空气中传播（喇叭到 MIC 的距离很近）                                       │
│                                                                                                 │
└────────────────────────────── MIC 再次拾取 ←────────────────────────────────────────────────────┘
```

每经过一轮循环，信号被增益放大一次（MIC 增益 33dB + DAC 音量 + PA 放大）。如果 **环路总增益 > 1**（即放大量 > 衰减量），信号将指数级增长，在几十毫秒内达到削波上限 → 产生刺耳啸叫。

虽然 echoTask 中尝试了简单的 AEC（回声消除）：`output = MIC1 - α × MIC3`，用 MIC3 参考信号去减除回声分量，但这种 **固定系数的减法** 远不足以消除声学反馈：
- 真正的 AEC 需要自适应滤波器（如 NLMS、频域自适应算法），系数需要实时收敛
- 喇叭到 MIC 的传递函数随环境（温度、角度、遮挡）实时变化
- 固定 α=0.78 只在特定条件下有效，稍有偏差就会失控

### 2.2 对比官方 Demo：我们做错了什么？

官方 Demo（`07_Audio_Test.ino`）的音频流程是 **完全串行的先录后放**：

```cpp
// Demo 的 Codec_LoopTask:
if (even & 0x01) {
    // 按键1 → 录音（此时喇叭没有在播放）
    codecport->CodecPort_EchoRead(audio_ptr, 192 * 1000);
}
if (even & 0x02) {
    // 按键2 → 播放（此时 MIC 的数据没有被送到喇叭）
    codecport->CodecPort_PlayWrite(audio_ptr, 192 * 1000);
}
```

**关键区别**：Demo 中录音和播放是**互斥的、分时的**：
- 录音时 → 只读 MIC，不写喇叭（PA 可能开着但 DAC 没有信号输入）
- 播放时 → 只写喇叭，不读 MIC（即使 MIC 在采集，数据也不会回到喇叭）

**我们的错误**：`echoTask` 同时做 `i2s_read()` + `i2s_write()`，将 MIC 实时数据直接送到喇叭，物理上形成了反馈环路。

### 2.3 音量小的原因

1. **软件增益缺失**：原始 `diagRecordThenPlay()` 直接将录音数据原样播放，没有做任何数字增益处理
2. **DAC 音量保守**：ES8311 REG32 设为 0xBF（0dB，unity gain），而 Demo 中 `CodecPort_SetSpeakerVol(100)` 通过 `esp_codec_dev` 框架映射到更高的增益
3. **MIC 模拟增益有限**：ES7210 MIC1 设为 33dB，麦克风灵敏度和距离导致原始信号幅度较小

## 三、调试过程和解决思路

### 3.1 调试的分步隔离法

面对"没有声音"或"声音异常"这类问题，核心思路是 **分段隔离验证**，将整条音频链路拆成独立的段，逐段确认：

```
段1: MIC → ES7210 → I2S RX → ESP32（录音链路）
段2: ESP32 → I2S TX → ES8311 → PA → Speaker（播放链路）
段3: 段1 + 段2 同时工作（全双工/回环）
```

**调试函数对应**：
| 调试函数 | 验证的链路段 | 方法 |
|---|---|---|
| `diagMicInput()` | 段1：MIC→ESP32 | PA 关闭，读 I2S RX，打印峰值 |
| `diagSpeakerOutput()` | 段2：ESP32→Speaker | 写 1kHz 方波到 I2S TX，听是否有声 |
| `diagRecordThenPlay()` | 段1 + 段2（串行） | 先录 2s，再播放 |
| `startEcho()` | 段1 + 段2（并行） | 实时回环 ← **问题在这里** |

通过这个分步验证，我们确认了：
- 段1（录音）正常：`diagMicInput` 显示有信号
- 段2（播放）正常：`diagSpeakerOutput` 能听到方波
- 段1+段2 串行（先录后放）正常：`diagRecordThenPlay` 能听到自己的声音
- 段1+段2 并行（实时回环）异常：`startEcho` 产生啸叫

结论：**问题不在硬件配置，而在软件架构（实时回环 vs 先录后放）**。

### 3.2 解决方案

参照 Demo 的模式，改为 **先录后放**：
1. 录音阶段：PA 关闭，纯粹采集 MIC 信号到 PSRAM 缓冲区
2. 增益处理：对录音数据做 16 倍数字放大（+24dB）
3. 播放阶段：PA 开启，将缓冲区数据写入 I2S TX

两个阶段物理上互斥，从根本上切断反馈环路。

### 3.3 以后遇到类似问题的快速定位方法

**原则：先隔离，再组合**

```
1. 画出完整的信号链路图（从输入到输出的每个环节）
2. 将链路拆分成可独立测试的段
3. 对每段编写最小测试代码，单独验证
4. 逐段确认后，按照 Demo 的方式组合
5. 如果组合后出问题，问题一定在"组合方式"上（时序、并行、反馈）
```

**常见陷阱清单**：

| 陷阱 | 表现 | 检查方法 |
|---|---|---|
| 声学反馈 | 啸叫 | 确认录音和播放是否同时进行 |
| I2C 写入失败 | 无声 | 回读寄存器验证 |
| I2S 格式不匹配 | 噪声/无声 | 确认 bit-width、format、MCLK 与 codec 一致 |
| DMA 残留数据 | 开头杂音 | 操作前调 `i2s_zero_dma_buffer()` |
| PA 时序 | pop 噪声 | 先设 DAC 音量 → 再开 PA；先关 PA → 再 mute DAC |

## 四、ES7210 和 ES8311 完整传输链路

### 4.1 硬件连接总览

```
                        I2C Bus (SDA=GPIO13, SCL=GPIO14)
                ┌──────────────┬──────────────────┐
                │              │                  │
            ES7210(0x40)   ES8311(0x18)        PCF85063/SHTC3
            (MIC ADC)      (Speaker DAC)       (RTC/温湿度)

                        I2S Bus (共享)
        ┌─────────────────────────────────────────────┐
        │  MCLK = GPIO16 (4.096 MHz, ESP32 APLL)     │
        │  BCLK = GPIO9  (1.024 MHz)                 │
        │  WS   = GPIO45 (16 kHz, = LRCK)            │
        │  DOUT = GPIO8  (ESP32 TX → ES8311 SDPIN)    │  ← 播放数据
        │  DIN  = GPIO10 (ES7210 SDOUT → ESP32 RX)    │  ← 录音数据
        └─────────────────────────────────────────────┘

        GPIO46 ──→ PA Enable（高电平有效）──→ 功率放大器 ──→ 扬声器
```

### 4.2 录音数据流（MIC → ESP32）

```
步骤1: 声波 → MIC1 振膜振动 → 产生微弱电信号（mV 级）
          │
步骤2: ES7210 模拟前端
          │  MIC 偏置电压 2.87V（REG41=0x70）使 MEMS MIC 正常工作
          │  PGA 可编程增益放大：33dB（REG43=0x1B）
          │  增益表: 0dB, 3dB, 6dB, ..., 33dB, 36dB
          │
步骤3: ES7210 内部 Sigma-Delta ADC
          │  模拟信号 → 数字 PCM 样本（16-bit, -32768 ~ +32767）
          │  过采样率 OSR=32（REG07=0x20），内部时钟 = 外部 MCLK
          │  高通滤波器去除直流偏置
          │
步骤4: ES7210 I2S 输出（SDOUT 引脚 → GPIO10）
          │  格式：I2S Normal (Philips)
          │  数据在 WS 下降沿后一个 BCLK 开始发送
          │  16-bit 有效数据左对齐在 32-bit slot 的高位：
          │    | sample[15:0] | 0000_0000_0000_0000 |
          │    |<---------- 32-bit I2S word -------->|
          │  Stereo: L 时隙 = MIC1, R 时隙 = MIC3
          │
步骤5: ESP32 I2S RX 控制器
          │  DMA 自动将接收到的数据写入环形缓冲区（3 × 512B）
          │  i2s_read() 从 DMA 缓冲区读取数据到用户 buffer
          │  每次读 512B = 64 帧 = 4ms 的音频
          │
步骤6: 用户缓冲区（PSRAM）
            384KB = 3 秒 × 128KB/s
```

### 4.3 播放数据流（ESP32 → Speaker）

```
步骤1: 用户缓冲区（PSRAM）
          │  软件增益：每个 16-bit 样本 × 16（+24dB）
          │  削波保护：clamp 到 [-32768, +32767]
          │
步骤2: ESP32 I2S TX 控制器
          │  i2s_write() 将数据写入 TX DMA 环形缓冲区
          │  DMA 自动从缓冲区读取数据，通过 I2S 总线发送
          │  DOUT(GPIO8) 上的数据在 BCLK 上升沿更新
          │
步骤3: ES8311 SDPIN 引脚接收 I2S 数据
          │  格式匹配：I2S Normal, 16-bit 数据在 32-bit slot 高位
          │  REG09 bit6=0: SDPIN unmute（接收数据有效）
          │
步骤4: ES8311 内部 DAC
          │  数字 PCM 样本 → 模拟电压
          │  DAC 过采样率 OSR=32（REG04=0x20）
          │  DAC 数字音量控制（REG32）:
          │    0x00 = -95.5dB (静音)
          │    0xBF = 0dB (unity)
          │    0xFF = +32dB (最大)
          │
步骤5: ES8311 模拟输出 → PA（功率放大器）
          │  GPIO46 = HIGH 时 PA 使能
          │  PA 将 ES8311 的模拟输出放大到喇叭驱动电平
          │
步骤6: 喇叭发声
```

### 4.4 I2S 总线时序细节

```
         ┌──── 一帧 (1/16000 s = 62.5 μs) ────┐
         │  L channel (32 BCLK)  │  R channel   │
         │                       │  (32 BCLK)   │
    WS:  ────┐                   ┌───────────────┐
              └───────────────────┘               └───

  BCLK:  ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─    ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
           └─┘ └─┘ └─┘ └─┘      └─┘ └─┘ └─┘ └─┘

  DATA:   X D15 D14 D13 ... D0 0 0 ... 0    (32 bits total)
           ↑
           WS 下降沿后 1 个 BCLK → 开始发送 MSB

  时钟关系:
    MCLK = 4,096,000 Hz（ESP32 APLL → GPIO16）
    BCLK = MCLK / 4 = 1,024,000 Hz（GPIO9）
    WS   = BCLK / 64 = 16,000 Hz（GPIO45，= 采样率）

  一个 BCLK 周期 = 1/1,024,000 ≈ 0.977 μs
  一帧 = 64 BCLK = 62.5 μs = 1/16000 s
```

### 4.5 I2C 控制总线

```
I2C 用于配置 codec 寄存器，不传输音频数据。

  ESP32 (Master)
    │ SDA = GPIO13
    │ SCL = GPIO14
    │ 频率 = 100kHz (Arduino Wire 默认)
    │
    ├──→ ES8311 (0x18): DAC 音量(REG32)、时钟(REG01-06)、格式(REG09/0A)
    ├──→ ES7210 (0x40): MIC 增益(REG43-46)、位宽(REG11)、电源(REG4B/4C)
    ├──→ PCF85063 (0x51): RTC
    └──→ SHTC3: 温湿度传感器

注意：I2C 和 I2S 是完全独立的两条总线。
  I2C = 控制面（寄存器读写，低速）
  I2S = 数据面（音频 PCM 流，高速）
```

## 五、最终修复总结

### 5.1 代码改动

| 文件 | 改动 |
|---|---|
| `AudioCodec.h` | 删除 `startEcho/stopEcho/startTone/diag*`，新增 `startRecordPlay()/stop()` |
| `AudioCodec.cpp` | 删除 `echoTask`（啸叫根源），新增 `recordPlayTask`（先录后放） |
| `Es8311Init.cpp` | 加详细中文注释 |
| `Es7210Init.cpp` | 加详细中文注释 |
| `MusicPlayerState.cpp` | 删除所有诊断调用，KEY2 触发录音 3s → 自动播放 |

### 5.2 关键设计决策

1. **先录后放（非实时回环）**：从物理上切断声学反馈环路
2. **录音时关闭 PA**：双重保险，即使 DAC 有噪声也不会被 MIC 拾取
3. **软件增益 16x（+24dB）**：弥补 MIC 信号幅度不足
4. **清空 DMA 缓冲区**：录音前和播放前都清空，避免残留数据导致的杂音
5. **PA 开关时序**：开启时先设音量再开 PA，关闭时先关 PA 再 mute DAC，避免 pop 噪声

### 5.3 经验教训

1. **先看 Demo 再动手**：官方 Demo 就是先录后放，不做实时回环。如果一开始就参照 Demo 的架构，就不会走弯路
2. **音频系统的第一原则**：如果 MIC 和 Speaker 同时工作且物理距离近，必须有成熟的 AEC 算法或物理隔离，否则必然啸叫
3. **分段隔离调试**：把链路拆成独立可测试的段，逐段验证，最后再组合
4. **寄存器初始化要有参考**：ES8311/ES7210 的寄存器配置直接照搬 Espressif 的 `esp_codec_dev` 驱动源码，不要自己猜
