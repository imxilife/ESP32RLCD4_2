# ESP32-S3 中 PSRAM 与 RAM 使用说明

## 1. 这份文档解决什么问题

这份文档用于快速回答以下几个问题：

- 什么是内部 RAM / SRAM
- 什么是 PSRAM
- 两者有什么区别
- 什么时候应该使用 PSRAM
- 什么时候不应该使用 PSRAM
- 在本项目里哪些场景适合放到 PSRAM

适用硬件背景：

- ESP32-S3
- 片上 SRAM：约 512KB
- 外挂 PSRAM：8MB
- Flash：16MB

## 2. RAM 和 PSRAM 的核心区别

### 2.1 内部 RAM / SRAM

内部 RAM 是芯片片上内存，特点是：

- 访问速度快
- 延迟低
- 适合实时性强的数据
- 容量相对较小

典型用途：

- 任务栈
- 中断相关数据
- 高频访问的小对象
- 驱动状态
- 实时控制逻辑中的关键变量

### 2.2 PSRAM

PSRAM 是外挂扩展内存，由 ESP32-S3 映射后供程序动态分配，特点是：

- 容量大很多
- 速度比内部 SRAM 慢
- 延迟更高
- 更适合大块缓冲区和大对象

典型用途：

- 音频缓冲
- 图片缓存
- 下载/上传大块缓存
- 临时大数组
- 大型容器或工作区

## 3. 一个简单理解方式

可以把两者理解成：

- SRAM：主工作台，快，但面积小
- PSRAM：旁边的大仓库，容量大，但取用慢一些

因此原则通常是：

- 小而频繁的数据，优先放 SRAM
- 大而不那么敏感的数据，优先放 PSRAM

## 4. PSRAM 不能替代什么

PSRAM 不是 Flash，也不是分区表的一部分。

它不能解决这些问题：

- 固件太大
- OTA app 分区太小
- `partitions.csv` 空间不够

这些问题属于 Flash / 分区问题，应通过下面方式解决：

- 调整 `partitions.csv`
- 缩减字体和资源
- 减少不必要库

一句话：

- Flash 不够：改分区、减代码、减资源
- RAM 不够：考虑用 PSRAM

## 5. 什么时候应该优先考虑 PSRAM

当你遇到以下情况时，应优先考虑把数据放到 PSRAM：

- 单个缓冲区很大
- 同时存在多个大缓冲
- 数据量大，但不要求极低延迟
- 运行时内存吃紧，而不是固件空间吃紧

典型例子：

- MP3 解码 PCM 缓冲
- 录音/播放双缓冲
- OTA 下载缓冲
- Web 上传过程中的大数据块缓存
- GUI 离屏缓存
- 大型字体/图片的运行时缓存

## 6. 什么时候不要优先放 PSRAM

下面这些内容通常不适合优先放到 PSRAM：

- ISR 直接访问的数据
- FreeRTOS 关键任务栈
- 高频率、小粒度、频繁随机访问的数据
- 强实时路径上的控制变量
- 某些 DMA 必须使用的专用内存

如果把这些数据放到 PSRAM，可能会出现：

- 响应变慢
- 时序不稳定
- 驱动行为异常

## 7. 在 Arduino / ESP32 项目里怎么使用 PSRAM

### 7.1 使用 `ps_malloc`

如果 Arduino-ESP32 已启用 PSRAM，最常见方式是：

```cpp
uint8_t* buf = (uint8_t*)ps_malloc(4096);
```

适合简单大块分配。

### 7.2 使用 `heap_caps_malloc`

更明确的方式是：

```cpp
uint8_t* buf = (uint8_t*)heap_caps_malloc(4096, MALLOC_CAP_SPIRAM);
```

这表示显式要求从 PSRAM 分配。

如果需要放在内部 RAM，也可以使用其他 capability 约束。

### 7.3 判空检查

无论哪种方式，都必须检查分配结果：

```cpp
if (!buf) {
    Serial.println("alloc failed");
}
```

- 大图片或位图缓存
- 字体渲染缓存

## 9. 判断一个对象该放哪的实用规则

可以用下面 4 个问题快速判断：

1. 这个对象大不大？
2. 它访问频率高不高？
3. 它对实时性敏不敏感？
4. 它失败时是否能接受降级？

经验判断：

- 大 + 不敏感：放 PSRAM
- 小 + 高频 + 敏感：放 SRAM

## 10. 常见误区

### 误区 1：PSRAM 越多越好，所有东西都放进去

不对。PSRAM 更像扩容区，不是所有数据的默认位置。

### 误区 2：把常量放到 PSRAM 能减少 Flash

不对。常量通常应放 Flash 的 rodata。  
PSRAM 解决的是运行时内存问题，不解决固件体积问题。

### 误区 3：分区表和 PSRAM 有关系

不对。`partitions.csv` 只管理 Flash 分区，不管理 PSRAM。

## 11. 当前项目的建议

结合当前项目状态：

- Flash 已通过扩大 OTA app 分区缓解
- RAM 当前占用较低
- PSRAM 暂时不是瓶颈，但未来会在以下方向继续有价值：
  - 音频
  - OTA
  - 大 UI 缓冲

因此当前建议是：

- 保持关键控制逻辑留在 SRAM
- 大缓冲继续优先放 PSRAM
- 如果后续引入图片缓存、网页缓存、AI 数据缓存，优先评估 PSRAM

## 12. 一句话总结

最实用的结论只有两句：

- SRAM 用来放“快而关键”的数据
- PSRAM 用来放“大而不那么敏感”的数据

如果不知道该怎么选，先问自己：

“这个对象是不是大块缓存，而不是实时控制变量？”

如果答案是“是”，那它通常就是 PSRAM 候选。
