# SD 卡问题复盘

## 结论

本次问题的直接结论是：

1. 当前硬件和官方示例实际使用的是 `SDMMC 1-bit` 模式，不是我们最初实现的 `SPI + CS` 模式。
2. 项目最初把 `GPIO21/GPIO38/GPIO39` 误当成 `MOSI/SCK/MISO` 来初始化 `SD.begin(...)`，导致一直在错误的协议栈上做排查。
3. 切换到和官方示例一致的 `SDMMC 1-bit` 初始化后，SD 卡识别与后续读写恢复正常，问题解决。

## 现象

最初在 Launch 页触发 `SD 卡测试` 后，识别阶段持续失败，典型日志如下：

```text
[SD] SPI init: SCK=38 MISO=39 MOSI=21 CS=17
[SD] Try mount at 4000000 Hz
[E][sd_diskio.cpp:199] sdCommand(): Card Failed! cmd: 0x00
[SD] SD.begin(...): FAIL
```

后续增加总线探针后，日志进一步表现为：

```text
[SD] idle-high probe (CS=HIGH): FF FF FF FF FF FF
[SD] card-selected probe (CS=LOW): FF FF FF FF FF FF
```

这说明卡在 `CMD0` 阶段完全没有响应，失败发生在最早的初始化握手阶段，而不是文件系统挂载或文件读写阶段。

## 排查过程

### 1. 先把问题拆成“识别”和“读写”

最开始 Launch 页入口直接跑整套 `mount + read/write + delete + listDir`，这会把“没识别到卡”和“识别后读写失败”混在一起。

因此先做了两件事：

- 把 SD 测试拆成 `Phase 1: Detect Card` 和 `Phase 2: Read/Write Validation`
- 让 `Phase 1` 失败时直接停止，不再继续跑读写测试

这一步的价值是先确认根因到底在“卡有没有起来”还是“文件系统/读写”。

### 2. 在 SPI 路线下补观测点

在当时的 SPI 实现里，补充了这些观测：

- `SD.begin(...)` 的成败
- `cardType()` 返回值
- SPI 预初始化时序
- `CS=HIGH / CS=LOW` 下的读回探针
- 多个频率降速重试

结果三档频率都失败，且 `CS=LOW` 时仍然全 `FF`。这说明：

- 不是因为频率太高
- 不是 FAT 文件系统问题
- 更像是协议模式或硬件连接方向就不对

### 3. 联查原理图

后续对原理图做了两轮核对：

- 一轮是确认 `GPIO21/GPIO38/GPIO39` 是否确实连到卡座
- 一轮是确认卡座附近是否有上拉、片选、未用脚处理

原理图里能确认到：

- `GPIO21`、`GPIO38`、`GPIO39` 确实连到卡座
- 卡座附近有 4 路 10K 上拉
- 图上还画了 `SDCS -> GPIO17` 的 SPI 片选链路

如果只看这里，很容易继续沿 SPI 方向怀疑 `SDCS`。

### 4. 对照官方示例发现关键分歧

真正把问题定性的，是对照官方 `06_SD_Card` 示例。

官方示例并没有用 Arduino `SD.begin(cs, spi, ...)`，而是：

- `sdmmc_host_t host = SDMMC_HOST_DEFAULT();`
- `slot_config.clk = 38;`
- `slot_config.cmd = 21;`
- `slot_config.d0 = 39;`
- `width = 1`

也就是说，官方实际使用的是：

- `CLK = GPIO38`
- `CMD = GPIO21`
- `D0 = GPIO39`
- 模式为 `SDMMC 1-bit`

这与 SPI 语义不同：

- `GPIO21` 在这里是 `CMD`，不是 `MOSI`
- `GPIO39` 在这里是 `D0`，不是 `MISO`
- `GPIO17 / SDCS` 在这条官方链路里根本不参与初始化

到这里，问题就清楚了：

> 我们不是在“正确硬件上调一个有问题的 SPI 驱动”，而是在“把一块按 SDMMC 用的硬件误当成 SPI 设备来驱动”。

## SD 卡驱动方式技术细节

这次问题之所以会绕远，本质上是因为“同一张 SD 卡”可以被两套不同的主机接口驱动，而两套接口的信号语义、初始化方式和软件栈都不一样。

### 1. SPI 模式

SPI 模式是把 SD 卡当成一个 SPI 从设备来使用，常见信号是：

- `SCK`
- `MOSI`
- `MISO`
- `CS`

在这个模式下：

- 主控通过 `CS` 选中卡
- 命令和参数从 `MOSI` 发给卡
- 卡从 `MISO` 回应
- 通常使用 Arduino `SD.begin(csPin, spi, freq)` 这一类接口

它的特点是：

- 接法简单，很多 MCU 都容易支持
- 软件上手快，调试门槛低
- 但吞吐通常低于原生 SDMMC
- 强依赖 `CS` 这条片选链路

这次最初的错误实现就是把板子按 SPI 来驱动：

- `GPIO21` 当 `MOSI`
- `GPIO38` 当 `SCK`
- `GPIO39` 当 `MISO`
- `GPIO17` 当 `CS`

但官方示例并不是这么用的。

### 2. SDMMC / SD 原生总线模式

SD 卡除了 SPI 模式，还支持原生 SD 总线模式。ESP32 系列里通常叫 `SDMMC`。

这个模式下最核心的几根线是：

- `CLK`
- `CMD`
- `D0`
- `D1`
- `D2`
- `D3`

和 SPI 的区别是：

- `CMD` 是双向命令线，不等于 `MOSI`
- `D0~D3` 是数据线，不等于单纯的 `MISO`
- 没有 SPI 意义上的 `CS`
- 主机通过 SD 协议直接和卡交互，不走 SPI 命令封装

ESP32 上常见的接口形态有：

#### `SDMMC 1-bit`

只使用：

- `CLK`
- `CMD`
- `D0`

特点：

- 接线最少
- 不依赖 `D1/D2/D3`
- 吞吐低于 4-bit，但通常仍高于 SPI
- 很适合资源受限或布线已经固定的项目

这次官方示例走的就是这种模式：

- `CLK = GPIO38`
- `CMD = GPIO21`
- `D0 = GPIO39`

#### `SDMMC 4-bit`

使用：

- `CLK`
- `CMD`
- `D0`
- `D1`
- `D2`
- `D3`

特点：

- 数据带宽更高
- 更适合大文件、高吞吐场景
- 需要更多引脚和更完整的布线
- 对布线质量、上拉、EMI 和时序更敏感

从协议角度讲，很多资料会把这类接口统称为 `SD mode` 或 `SDMMC mode`。

### 3. 1-bit、2-bit、4-bit 的差别

从这次项目语境里，最重要的是理解“位宽”代表一次时钟能并行传多少数据。

- `1-bit`：一次时钟走 1 根数据线
- `4-bit`：一次时钟走 4 根数据线

位宽越宽：

- 峰值吞吐越高
- 引脚占用越多
- 布线和硬件约束越强

在常见 MCU/ESP32 场景里，最常用的是：

- `SPI`
- `SDMMC 1-bit`
- `SDMMC 4-bit`

“2-bit”在很多实际项目里不是主流工作模式，至少在当前这块板子和官方示例里没有采用，也不是这次问题的重点。

### 4. 两种方式的软件驱动差异

#### SPI 方式如何驱动

SPI 方式通常分两层：

1. 初始化 SPI 控制器和 `CS`
2. 调 `SD.begin(...)` 或等价库接口完成卡识别与 FAT 挂载

典型特征：

- 代码里会出现 `SPIClass`
- 会显式配置 `CS`
- 会使用 `SD.open()/SD.exists()/SD.remove()`

如果 `CS` 不通、`MISO` 无响应、SPI 时序不对，常见表现就是：

- `CMD0` 失败
- `cardType()` 取不到
- 一直显示未检测到卡

#### SDMMC 方式如何驱动

SDMMC 方式同样分两层：

1. 配置 `host` 和 `slot`
2. 通过 `esp_vfs_fat_sdmmc_mount(...)` 或等价接口完成挂载

典型特征：

- 代码里会出现 `sdmmc_host_t`
- 会配置 `slot_config.clk/cmd/d0(/d1/d2/d3)`
- 文件访问常通过 VFS 挂载点，例如 `/sdcard/...`

这次最终修复采用的就是这种方式。

### 5. 为什么这次会误判

这次最容易误判的点在于：

1. 原理图上同时出现了 `MOSI/SCK/MISO/SDCS` 这类命名，表面上很像 SPI
2. 但官方示例实际用的是 `CLK/CMD/D0` 语义和 `sdmmc_mount` 软件栈
3. 如果只看“GPIO 号对得上”，不先确认协议模式，就很容易把 `CMD` 误读成 `MOSI`，把 `D0` 误读成 `MISO`

因此，这次真正需要先回答的问题不是“引脚对不对”，而是：

> 这块板子的 SD 卡，在软件上到底应该被当成 SPI 设备，还是 SDMMC 设备？

## 根因

根因可以分为两层。

### 直接根因

项目里的 SD 驱动实现选错了协议模式：

- 代码实现：`SPI + SD.begin(...)`
- 板级真实用法：`SDMMC 1-bit + esp_vfs_fat_sdmmc_mount(...)`

因此从初始化第一步开始就走错了。

### 过程根因

出现误判的原因主要有三点：

1. 过早从已有代码出发，把 `21/38/39/17` 解释成 SPI 引脚，而不是先用官方示例校准协议模式。
2. 原理图里同时存在 `SDCS` 这条链路，容易让人先入为主地认为该接口应该走 SPI。
3. 初始测试没有把“识别”和“读写”拆开，导致排查入口不够聚焦。

## 最终修复

最终采用的修复方案是：

1. 放弃 SPI 实现，不再依赖 `SD.begin(...)`
2. 将 `lib/device/sdcard/SDCard.cpp` 切换为和官方一致的 `SDMMC 1-bit` 初始化
3. 保持原有 `SDCard` 封装接口不变，只替换底层实现
4. 将 MP3 读取也切换到同一挂载点 `/sdcard`，避免测试通过但播放器仍走旧后端

修复后的关键点：

- `begin()` 通过 `esp_vfs_fat_sdmmc_mount(...)` 初始化
- 文件读写通过 `/sdcard/...` 路径访问
- 测试仍保留“先识别，再读写”的分阶段结构

## 经验教训

### 1. 先确认协议模式，再确认引脚

同一组物理引脚在不同模式下语义完全不同：

- SPI：`MOSI / MISO / SCK / CS`
- SDMMC 1-bit：`CMD / D0 / CLK`

只确认“引脚号对得上”是不够的，必须先确认协议模式。

### 2. 官方示例优先级高于自我推断

遇到板级外设问题时，优先检查：

1. 官方示例到底用了什么 API
2. 官方示例到底用了什么模式
3. 官方示例到底用了哪些引脚

不要先按熟悉的驱动模型套进去。

### 3. 排查要先切断问题耦合

像 SD 这种问题，必须先拆成：

- 卡识别是否成功
- 挂载后读写是否成功

否则日志很多，但信息密度低。

### 4. 嵌入式排查要同时准备多条思路

同一个现象，至少同时保留三种假设：

- 软件协议模式错
- 硬件连接/上拉/供电错
- 使用方式与官方样例不一致

不要在第一条解释上停太久。

## 后续建议

为避免类似问题再次出现，后续涉及板级外设时建议固定采用以下顺序：

1. 先找官方示例或官方 BSP
2. 再对照原理图确认连接关系
3. 再决定驱动模式和封装接口
4. 最后才做读写、性能或稳定性优化

如果未来再遇到 SD 异常，优先先看：

- 当前代码是否仍在使用 `SDMMC 1-bit`
- 挂载点是否仍是 `/sdcard`
- 上层模块是否绕过 `SDCard` 直接访问了其它存储后端
